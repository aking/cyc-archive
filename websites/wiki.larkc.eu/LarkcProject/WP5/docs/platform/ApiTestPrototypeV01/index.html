LarKC Plug-in API - Prototype Version 01

Barry Bishop, Mick Kerrigan - 9th September 2008

1. Introduction

This article contains the developer's notes relating to a test implementation of the first draft of the LarKC plug-in API.

A first draft of the API was published shortly after the Ljubljiana workshop. This API was intended to be as simple as possible and yet still have enough features to allow plug-ins to be developed that could achieve the most basic goals of the LarKC platform. In order to verify that this API was sufficient, Mick and Barry created a number of simple plug-ins (Identify, Abstract, Select, Reason) and glued them together with a super-simple Decide component.

The results so far have been encouraging. Even this simple prototype allows the user to execute SPARQL queries against RDF triples stored throughout the web.

The rest of this article describes the plug-in components created, how the API was modified and what issues have arisen that require further exploration, discussion and agreement between the work package 5 partners.

It is planned to add an ORDI/OWLIM storage component in the next few days, after which development work on this version will be frozen. This is so that we have a version of the API, test implementation, source code, notes, design decisions, etc, that we can look back on. Future work to evolve the API further will be conducted on forks of this code-base: platform_v02, platform_v03, etc (see the 'next steps' section).

If you want to take this code and conduct your own experiments then you are strongly advised to fork this code-base and place the code in a suitably named folder in the subversion repository, e.g. platform_barry_experiments23.

The source code for this revision of the API and prototype plug-ins can be found here:

https://svn.gforge.hlrs.de/svn/larkc/trunk/platform_v01

2. How to Run The Demo

If you wish to try out the web-interface then you will require a working installation of tomcat:

  • Copy the platform_v01/web-demo/larkc folder in to the webapps folder of your tomcat instance

  • Start your browser and point to http://localhost:8080/larkc

There is a more basic way of testing out the prototype by running demo/eu.larkc.demo.ExecuteSparqlQueryAgainstWebResources.main. The query to execute is hard coded in this java file.

3. Implemented Plug-ins

In pipe-line order, the following plug-ins were created:

3.1. Identify

Two components were created to wrap the sindice RDF indexer website http://sindice.com/developers/querylanguages.

  • SindiceKeywordIdentifier

  • SindiceTriplePatternIdentifier

The first of these uses keywords found in the sparql query to call the keyword search facility of sindice. The second uses triple patterns extracted from the WHERE clause of the SPARQL query to call the triple pattern search facility of sindice.

Sindice allows a variety of formats to be returned, the most straightforward of which is 'atom'. However, it was discovered that there was a large overhead in using a standard XML parser to parse the result and so there is a simpler, but faster parsing step in the component.

Sindice returns the URLs of documents on the web likely to contain RDF data related to the search parameters. These URLs might identify RDF/XML documents or RDF embedded in other formats/documents, e.g. HTML.

Conceptually, each URL identifies a collection of RDF triples located somewhere on the web and each URL is returned in a concrete class (TripleSetUrl) that implements the TripleSet interface.

3.2. Transform

The API currently defines two transformer interfaces:

  • DataSetTransformer

  • QueryTransformer

The first of these is not utilised in this prototype.

However, QueryTransformer is implemented by two classes:

  • SPARQLToTriplePatternQueryTransformer - transforms the SPARQL query in to a collection of triple pattern queries
  • SPARQLToKeywordQueryTransformer - transforms the SPARQL query in to a collection of keyword queries

These two implementation are specifically for input to the two sindice identify components.

3.3. Select

The input to this stage is a collection of resources (triple sets) and the output is a sample of the available data (a single triple set). Again, two simple implementations were created:

  • FirstDataSetOnlySelecter - which simply returns the first resource (triple set)

  • GrabEverythingSelecter - which combines all the supplied resources

The first of the above returns the first triple set unaltered. However, the second implementation needs to actually fetch the RDF triples identified by each URL in order to combine them. Perhaps this could be a bottleneck when large amounts of data are involved.

When many resources are identified, the GrabEverythingSelecter was taking progressively longer to retrieve each document from the web. Therefore the decision was taken to retrieve and parse each resource on a separate thread and combine the results when the last thread finishes. Perhaps QoS parameters could be applied here, so that each thread will time-out after some maximum limit is reached.

3.4. Reason

The only Reason component implemented was a SPARQL query engine based on the jena framework http://jena.sourceforge.net/.

  • SparqlQueryEvaluationReasoner

This executes the input SPARQL query against a jena model object containing all the triples output from the Select stage.

3.5. Decide

The simplest of all possible decide components was implemented that executes a single pass of all supplied plug-ins:

  • SimpleConfigurableOnePassDecider

This decider calls each plug-in (once) in turn: Identify, Transform, Select, Reason and returns the results.

4. Utility/Support Classes

The following classes were implemented, grouped by the interface that they implement:

  • TripleSet

    • TripleSetContainer - in memory collection of triples - not used

    • TripleSetUrl - a wrapper for a URL, but will actually fetch and cache the remote RDF data when requested

  • VariableBinding

    • JenaVariableBinding - uses a jena result set object to hold the results of a query

  • KeywordQuery (extends Query)

    • KeywordQueryImpl - a keyword query, basically just a list of keywords or tokens.

  • SPARQLSelectQuery (extends SPARQLQuery, extends Query)
    • SPARQLSelectQueryJena - implemented using a jena query object

  • TriplePatternQuery (extends Query)

    • TriplePatternQueryImpl - a collection of triple patterns (with wild-cards)

The following utility classes were created:

  • HttpUtils - Get and Post methods.

  • RdfUtils - Methods to extract RDF triples from a document identified by URL.

5. API Modifications Required

The following interfaces were added:

  • Triple - a generic interface for a single triple. In the end, we found that we didn't actually need this.

  • SparqlSelectQuery - a further spcialisation SPARQL query, but does not yet have any methods - possibly remove this.

  • TriplePatternQuery - an interface for queries about patterns of triples - needed for input to the sindice identify plug-in that uses triple patterns.

Interestingly, the plug-in interfaces were not modified at all.

6. Issues Arising

Observation: Combining resources from web locations is inherently expensive and time consuming. It would be interesting to investigate applying a time limit to fetching each resource (QoS) combined with 'anytime' behaviour so that even if the entire resource could not be read, at least some of it would be downloaded in the time available.

Observation: There seems to be a strong coupling between identify and storage components. For example, if sindice is used then resources will be identified by URL, which will be wrapped in a URL-based triple set class. However, if (for example) an identify stage selected one of several ORDI/OWLIM triple stores (or maybe a named graph in a triple-store) then the corresponding concrete triple set class will likely be specialised for addressing/accessing ORDI/OWLIM.

Observation: Some more design work is required for the classes that implement TripleSet. The idea is that a triple set can be identified by URL, or be in memory, or be in an OWLIM instance, or anywhere else, but we should try to make them appear the same by hiding them in a concrete class that implements TripleSet and is appropriate for the particular storage medium, location etc. More importantly, by doing this we can defer dereferencing the triple-set object for as long as possible and possibly avoid doing so completely (e.g. executing a query remotely where the triples are located). However, as we found in GrabEverythingSelecter, sometimes one needs to know what one is dealing with in order to proceed, e.g. if one wants to merge triple sets it is easier to do if they are all stored/represented in the same way. There seems to be too much "if( tripleSet instanceof XXXX) ...". Perhaps the only important factors are:

  • Location: triples in memory, locally stored, remotely stored
  • Iterable: can the triples be iterated over
  • Merging: Is it cheap or expensive to combine two triple sets? What properties can I expect for the resulting triple set.
  • Queriable: Does this triple set support a direct SPARQL query?

Observation: Transformation seems to happen at several and (in the future) at potentially many different points during execution:

  • Transforming the user query in to a form accepted by an identify component (e.g. SPARQL to keywords for sindice)
  • Transforming identified data in to RDF (not sure about if we will ever do this)

There are also occasions when data is not altered, but its location/data-structures are changed, e.g.

  • De-referencing of URLs in the select component (triple sets wrapping URLs become triple sets containing triples for merging)
  • De-referencing of URLs in the reason component for the same reasons as 'select'
