  • Finalization of Platform Documentation (all)
    • Missing input:

        • Alpha Urban LarKC. Check with Emanuele whether they like to have it here.

      • Section 4.5 Deployment and development using Netbeans (Luka, Cyc) – if we don’t have it ready “asap”, i´d remove it completely for this version
        • OUT
      • Section 4.6.2 and subsequent: Developing Transformer (Luka?) – Are we going to include instructions on every plug-in type? NO. Luka to write a general section on how to write plug-ins (independent on the type).
      • Caching features (Spyros) – Caching will not be included in this release. Then we should mention it in the next steps chapter. If some limited-caching features are being included, pls add some text about it in sections 4.6 and 4.7.
      • Document must be submitted for quality assessment this week!! ALL contribute to the assigned sections.
  • Upcoming release preparation (Alexey)
    • release candidate branch created. Trunk, excluding unnecessary/inmature features.

    • removed Demos components. Only stable components should be included in the release. Exemplary components, such as the Alpha Urban LarKC. Plus other exemplary components which add value to external users, such as the ?ConfigurableSimplePipeline.

    • configurablesimplepipeline works. We should include it. Alexey will try it. Contact Barry in case of problems. Some instructions in the early adopters tutorial. We should update them and include them in the documentation
    • JEE and JavaGAT remote execution will not be included in this release. Currently working on the “distributed Data Layer”, which will support the remote execution.
    • Michael: Plug-in developers should take care of their own releases (they are in separate projects within the release package).
    • Beside the “official” releases, we should make the intermediate features available to other WPs, e.g. WP6, in order to allow them to develop their workflows. Freeze branches as new features are being implemented (e.g. caching, flow controller,…).
  • Discussion on “Multiple queries, same workflow” + “Stateful plug-ins”
    • orthogonal to the flow controller.
    • to answer a query, larkc create a container of everything, including queues,…
    • multiple concurrent queries executed in one single workflow is not an easy task.
    • however, plug-ins executed concurrently in different workflows may be useful in some cases (when overhead to deploy a plug-in, e.g. in a remote resource, it very high compared to the total execution time). Different workflows using same plug-in instance, which is already deployed. For this purpose, we need stateless plug-ins.
    • Priority 1: we should concentrate on doing plug-ins stateless. Afterwards, we may look for a use case needing the feature above (1 plug-in instance, several workflows)
    • Getting plug-in stateless: use Context parameter. Method createContext() will be used by the plug-in manager. Barry to implement a first example. After this, other partners will take that as example to adapt other plug-ins.
  • Caching status (fast update) (Vassil, Spyros)

    • Vassil implemented sensible equals methods for ?InformationSets.

    • Spyros: I have implemented a ?CachingPipeline, which caches query results. In order to use it, it is enough to wrap any Pipeline with it. I still need to perform some tests and document it. I have also implemented caching for individual components, but I would not include this on the current release, since a Workflow is used only once and then thrown away, thus caching at plugin level makes no sense (for now).

    • @Alexey, what should I do with the ?CachingPipeline, should I commit it? Not in this release. It´s not sufficiently tested.

  • Flow Controller status (fast update) (Florian)
    • Florian will commit the code to the svn. Barry will remind him.
