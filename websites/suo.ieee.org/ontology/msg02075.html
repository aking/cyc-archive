Well, that is a pretty poor criticism.  Far from being a common concept

>
> (forall ?x (=> (car ?x) (has-wheels ?x)))
>
> were lambda expressions largely because they did involve variables.
> Please explain my error.

The expression above is simply a sentence in a particular first-order
language.  A lambda expression is a property-denoting (or function-, or
relation-denoting, depending on the flavor of lambda calculus)
expression involving an abstraction operator, usually written as a
lambda.  Thus, for example, in the typed lambda calculus, the expression
[lambda x (exists y) Rxy] would denote the property of being something
that bears the relation R to something, or, viewed extensionally, the
set of all things e such that, for some object e', the pair <e,e'> is an
element of the relation R.  (I'm mingling use and mention here a bit,
but never mind.)

Or close to it.  There is a proposal before the KIF WG that provides a

No, I think you are confusing giving with an instance of giving.
Granted, an instance of giving is an activity.  But giving is an
abstract relation -- the relation that three things a, b, and c stand in
when a gives b to c.  You distinguish between, say, the redness of this
apple (i.e., an instance of redness) and redness per se, don't you?
Same distinction applies with giving, don't you think?

> > The typed lambda calculus is basically just omega-order logic
>
> MW: Omega-order logic? You lost me.

Well, there's second-order logic, in which there are quantifiers range
over properties (or set) of individuals (among other things),
third-order logic in which there are quantifiers that range over
properties of properties of individuals (among other things), and so on.
Omega-order logic has quantifiers that range over all the things that
all the nth-order logics range over, for all n.
