>Mike, et al.
>
>But there seem to be some options about what the individual sibling
>relations are in the real world (is this possible if there is a real world?)
>
>As Fritz Lehmann has also noticed, there seem to be two camps using the same
>word (tuples) to denote different things.
>
>The logical view is that they are tuples - as Pat Hayes wrote:
>Close. The relation is a *set* of triples: all the triples that the
>relation holds true of. So for example if A sent Bill to B , then the
>triple <A, Bill,B> is one of the triples in the "billing by post"
>relation-set. Of course there are probably others in the same set.
>Every time someone gets billed, that defines another triple in the
>relation-set. Think of this set of triples as God's database of all
>the billings that ever happened or ever will happen. (It has to be
>God's since it might well be infinite.) As far as logic is concerned,
>this (potentially infinite) set of triples *is* the relation in
>question.
>
>The relational database view is that they are a different kind of tuple -
>where, somehow the tuple also carries content.

Wait a minute. You think that a triple of actual things (people and 
messages, say) doesnt carry content? How much more concrete could 
content get than actually being some pieces of the world? Its like 
pointing to the actual thing instead of describing it.

>We can distinguish them by their identity conditions. If we swap the columns
>in relational database we do not change the identity (or meaning) of the
>tuple.

If you swap the entries in one tuple in a table (without swapping the 
columns) you do change the meaning, though; thats the comparison you 
should make. This comparison is a bait-and-switch, since when you 
swap those 'columns' you are also swapping the implicit labellings of 
the columns. You can do that to mathematical relations as well, if 
you allow them to include that much extra structure.

>If we swap positions in a tuple e.g. <A, Bill, B> to <B, A, Bill> we
>do. Similarly while in logic a tuple can belong to a number of relation
>classes, it is not clear that this is so for the relational view.

So two things can only have one relation in a RDB ?

>One advantage of the logic view is that relations can be treated like
>properties - and there can be a simple taxonomy of relations. This is more
>difficult to achieve under the relational view. However, the logic view does
>lead to the seemingly odd conclusion that for every relation class and
>particular relation there also exists its inverse.

No, it doesn't. This is a delicate point but an important one (since 
it will be central in SUO-KIF, which can quantify over relations). 
Saying what a relation is does not automatically say that all things 
like that must be relations.  If every set of tuples was a relation, 
then we would have a genuine higher-order logic; but we do not make 
this assumption (though it is, I concede, often made by 
mathematicians.) So it is quite possible for a relation to exist but 
its inverse not to, or at any rate not to be considered a relation in 
the universe of all relations.
