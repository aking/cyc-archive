David,

> There seems to be a need to build the SUO as an artifact,
> attach comments and alternate formulations to any given sub-part of it,
> and then be able to see if the new artifact with the changes is
> more 'capable' than the old one.

I suggest we may already have a framework providing much of this capability,
in the SUO Dictionary -- to make it clear that it is intended to be used for
this purpose, I propose to rename it "SUO Common Ontology Development
Architecture (Coda)". Otherwise, people may still think it is just a
glossary, to be developed and used as an appendix or afterthought -- rather,
I hope that SUO members will find it a useful tool in the development of
SUO. It can support collaboration in development of one or more ontology
efforts, i.e. "merged ontology", "4D", "IFF", etc.

SUO Coda addresses the following requirements:

- open to input from multiple contributors
- a contributors input is easily published on the web, directly by the
contributor
- readers can view any contributor's work
- contributors can only edit their own contributions
- teams of people can collaborate to develop consensus definitions, or agree
to disagree
- multiple ontology efforts / directions can be pursued in parallel
- support for namespaces within each ontology effort
- openness to different notations, and different levels of formalization, to
facilitate flexibility in exploring new ideas, as well as rigorous
formalization of ideas.

Robert Kent has begun using this framwork and you can see the contributions
he has provided at:

https://www.quickbase.com/db/6urbwpxk?act=q&qid=1

>
> Jon Awbrey talked about the need to run a really merciless
> logic checker over it.  I expect that we will need to have ways
> of defining and re-defining the logic checker as well as the SUO.

Using the Quickbase API, it should be possible to write a Perl program that
would export information from SUO Coda, which could then be input into a
logic checker.
