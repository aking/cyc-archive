Dear Pat,

I know you (I) need it, the question is whether I can explain it adequately.
> >
> >The only thing not mentioned here that is important (in my opinion)
> >is a usage tree for when objects that are "defined" in other 
> statements
> >are used in this one. E.g.
> >(
> >REFNAME: Extensional-relation
> >USES: sequence-extensionality
> >...
> >)
> 
> I can't quite see the point of this, since one can see what each 
> axiom uses just by looking at the axiom.

MW: What you can see in the axiom is the names of the other constants that
are used in it, not where to find them. The vision I have is that if you
have a 
library of axioms in different stores (I leave open whether this is a
database or a file system) then what you have are the names of the axioms,
not the names of the constants defined in those axioms. If you have
something like a USES: statement then when you identify that for some piece
of reasoning you need some particular collection of axioms, you can
automatically pull in all the other ones that are relevant.

MW: The alternative as I see it is that for any piece of reasoning you have
to pull everything in. I expect everything to be quite large, eventually.

MW: This is an structural/engineering/implementation issue not a conceptual
one. However, one of the things we should take care of is that the SUO can
be used effectively and efficiently and in a standard way.

MW: In ISO TC184/SC4 there is a workgroup that look specifically at how
to implement EXPRESS based models in a standard way, with a standard API
etc.
