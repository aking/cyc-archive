Dear John,

Yes but ...

> 
> Matthew West wrote:
> 
> >>The only thing not mentioned here that is important (in my opinion)
> >>is a usage tree for when objects that are "defined" in 
> other statements
> >>are used in this one. E.g.
> >>(
> >>REFNAME: Extensional-relation
> >>USES: sequence-extensionality
> 
> Pat Hayes replied:
> 
> >I can't quite see the point of this, since one can see what each 
> >axiom uses just by looking at the axiom.
> 
> I agree with Matthew that it is desirable to have a display
> of what is used and referenced by every axiom in a module.
> But I also agree with Pat that such information can be
> obtained by looking at each axiom.
> 
> Rather than burden every axiom with extra declarations,

MW: Actually the burden is rather small, and indeed a useful 
discipline when creating modules. Also I would expect an warning to
be generated if there were constants for which there were no defining
axioms (not quite sure how you work that out though).

> I would suggest that KIF (or any similar language) be supported
> with tools that analyze all the reference patterns in a module
> or collection of modules and construct such defines/references
> graphs.

MW: In which case we need to define implementation standards for this.
Again, you seem to be assuming that the SUO is used in on all or nothing
basis, whereas I am assuming more lightweight uses where necessary parts
of the SUO are drawn down for use.
> 
> Those graphs could be used by various kinds of tools.  For
> example, if you wanted to extract a subset of axioms from
> a module, it would be important to include all the other
> axioms they reference.

MW: First, I would expect that modules would be taken on an all or 
nothing basis, if some bits are irrelevant, then nothing is lost,
if some bits are wrong, then they shouldn't be in the same module.

MW: Where I think tools would be useful is when module names and 
content are changed. You could then do an impact analysis to see
what is affected by the changes you have made.

MW: To me this discussion means that we need to pay some serious
attention now as to how we think the SUO is going to be implemented
and used. My own assumption is that it will be held within a 
relational DBMS, with a reasoning engine sitting on top. I expect
others have different ideas. However, a number of questions arise:

 - What is the data model for the database?
 - What is the spec of the API?
 - What is the spec of the reasoning engine?
 - is there more than one for any of the above? (probably)
