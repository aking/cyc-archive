All,

Since hypergraphs seem to be a convenient model for relations in an
ontology, I give here another version of hypergraph (see
[http://suo.ieee.org/ontology/msg02252.html]). In addition, I define a
morphism for such hypergraphs, getting a category of hypergraphs. We assume
an "adequate" (possibly denumerable) collection of indexing "variables".
Indexing variables are used for naming the participants of relations. Tuples
are important in what follows.

For any set A, a tuple t of A-elements is a collection of (possibly
duplicate) A-elements that are indexed by variables. The indexing elements
of t form a subset of variables. More precisely, a tuple t = [arity(t),
pr(t)] is a pair consisting of a subset of variables arity(t) called the
arity or valence of t and a map (function) pr(t) : arity(t) -> A called the
projector of t. We use the exponential notation t \in A^arity(t) for tuples.
Although tuples are actually maps, we often use the tuple notation t = [tx |
x \in arity(t)] for a tuple. In particular, if we are using positive
integers as indexing variables, then we often use the notation t = [t1 ...
tn] for a tuple with arity {1 ... n}.

Let tuple(A) denote the collection of all A-tuples. For any function f :
A -> B, there is a tuple function tuple(f) : tuple(A) -> tuple(B), define by
composition:
    arity(f(t)) = arity(t) and pr(f(t)) = pr(t) . f;
that is, if t = [t1 ... tn] then f(t) = [f(t1) ... f(tn)].

A hypergraph is a graph in which generalized edges (called hyperarcs)
connect multiple nodes. Formally, a hypergraph A = [node(A), edge(A), dA]
consists of
    (1) a set of nodes node(A),
    (2) a set of hyperarcs edge(A), and
    (3) a vertex function dA : edge(A) -> tuple(node(A)) that maps a
hyperarc e \in edge(A) to its associated vertex dA(e) \in tuple(node(A)).

A (strict) hypergraph morphism f = [node(f), edge(f)] : A -> B consists of a
node function node(f) : node(A) -> node(B) and an edge (hyperarc) function
edge(f) : edge(A) -> edge(B) that satisfy the hypergraph morphism condition
expressed in terms of vertex functions as follows:
    (4) edge(f) . dB = dA . tuple(node(f)).
Or, using more intuitive notation, if e \in edge(A) is a hyperarc of A with
vertex [a1 ... an], then edge(f)(e) \in edge(B) is a hyperarc of B with
vertex tuple(node(f))([a1 ... an]) = [node(f)(a1) ... node(f)(an)]. There
are more relaxed versions of hypergraph morphism, where (4) is just set
inclusion.

The category Hypergraph has hypergraphs as objects and (strict) hypergraph
morphisms as arrows (morphisms). The underlying edge functor edge :
Hypergraph -> Set maps a hypergraph to its edge set and maps a hypergraph
morphism to its edge function. Same for nodes.

For any n-ary KIF relation
    (relation rel)
    (signature rel cls1 ... clsn)
a relational expression
    (rel ent1 ... entn)
can be modelled as a hyperarc with vertex function mapping this to the
entity tuple
    [ent1 ... entn].

So the relevant (instance) hypergraph here would have entities as its nodes
and relation(ship)s as its edges (hyperarcs) -- all at the level of
instances. The types 'rel' and 'cls1' ... 'clsn', and the signature
relation, would be represented in another (type) hypergraph.

By regarding expressions as relations, hypergraph morphisms, in possibly a
more relaxed version, could serve as a suitable base for the definition of
the "meaning-preserving translations".
