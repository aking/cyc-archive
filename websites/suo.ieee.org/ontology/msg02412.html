> MW:  The question is, why and when might this be useful?
>      The answer is that many n-ary relations are overconstrained.
>      I.e. they are not a good model because there are many cases
>      they do not support.
> 
> MW:  This is particularly so when activities are represented
>      by relations.  Take giving as an example.  Using a relation
>      to represent this does not only require that there is a giver,
>      a receiver, and a gift, but it requires that there is not more
>      than one giver, receiver, and gift.  Whilst it is true that
>      there are some giving acts for which this constraint holds,
>      it is not true for all giving activities.  And whilst you
>      could define relations for all the possible combinations
>      of different numbers of givers, receivers, and gifts,
>      this is not particularly practical.
> 
> MW:  So how does changing the representation of an activity
>      from a relation to a set and some binary relations help?
> 
> MW:  The answer is that it removes the constraints.
>      Actually it removes both the minimum constraints
>      and the maximum constraints.
> 
> MW:  Even removing the minimum constraint can be useful
>      from an epistemological viewpoint.  So whilst in
>      an act of giving ontologically we know that there
>      must be at least one giver, one given and one gift,
>      do we necessarily have knowledge of each of these
>      for any giving activity?  More particularly do we
>      always have to care?
> 
> MW:  Example:  Steve Redgrave was given his 5th gold medal in
>      rowing at the Sydney Olympics.  Do we care who gave it to him?
>      Would we wish to be prevented from recording anything about this
>      giving act if we could not identify the person who gave 
> it to him?
>      I think not.
> 
> MW:  Of course there are times when you do want to require some 
>      information.  Explicit axioms can be used to support this.
>      For example, if you do not have a product ordered and
>      a customer at least, then there is no purpose to
>      accepting only partial information as an order.

MW: Now you are just being silly. It doesn't matter whose definition
it is or what language it is in as long as it works. Therefore
the SQL above is sufficient definition of the relational
operation I refer to i.e.

      SELECT AND_Argument1.[Arg1-Value],
             AND_Argument2.[Arg2-Value],
             AND_Result.[Result-Value]
      FROM
        (AND_Argument1
           INNER JOIN AND_Argument2
             ON AND_Argument1.Row = AND_Argument2.Row)
           INNER JOIN AND_Result
             ON AND_Argument1.Row = AND_Result.Row;

Loosely translated into English this says:

Join AND_Argument1, AND_Argument2 on the column row.
Join the result to AND_result on the column row.
From the result select the columns 
Arg1-Value, Arg2-Value, Result-Value.

In any case, this is only required to support your view, it
is not a part of the underlying information.
