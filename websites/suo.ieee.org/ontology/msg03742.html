4.3.2.3  Mosaic Paradigm:  Twin Tables for Every Law

I now offer a specific proposal for beginning to deal with the
above-noted needs and problems, addressed to the framework of
a logically ordained relational database system.  This "mosaic"
or "twin tables" scheme can be related in the following manner.

Envision two tables for every relational concept, one table for
the positive examples and one table for the negative examples,
or the "counterexamples", of the concept.  It is useful to view
this scheme as analogous to drawing a circle in a venn diagram,
where each closed figure determines two complementary regions
in the universe of discourse.  Two points need to be noted:

1.  This design cannot be allowed to acquire the kind of dishonesty
    that is commonly associated with keeping a double set of books --
    there are semantic integrity constraints that must be followed
    in order to avoid shear duplicity.  These semantic constraints
    correspond to one set of axioms for the classical propositional
    calculus, and will be specified below.

    Nota Bene.  In logic, the adjective "classical" is used by way
    of specifying that the principle of double negation, ((x)) = x,
    or, equivalently, the principle of excluded middles, ( x ,  (x)),
    is invariably observed.

2.  There are good reasons for maintaining a separate and well-defined
    syntax of terms and expressions for "talking about" tables and their
    structural relationships, independently of a need to create tables or
    to enter data, and prior to any decision about what kinds of relations
    ought to be actualized.  It ought to be obvious that this "Twin Table"
    scheme has the following consequences:  One table or other in each pair,
    T or (T), will seldom be actualized.  Indeed, the explicit actualization
    of both tables can only happen when we are working with a finite universe
    of discourse, or when we need to select complementary subsets of a given
    finite set.

Picture each pair of tables as in the following example,
where T is the 1-column table for the concept "Area Code",
and (T) is the 1-column table for the concept "Non Area Code",
that is, "-(Area Code)-".  For simplicity, we may imagine for
the moment that we are working with a universe of discourse
X = {triciphers} that is limited to 3-digit decimal codes.

o-------------o-----o         o-------------o-----o
|  Area Code  |  #  |         | (Area Code) |  #  |
o-------------o-----o         o-------------o-----o
|             | 806 |         |             | 000 |
|             | 906 |         |             | 001 |
|             | 202 |         |             | 010 |
|             | 313 |         |             | 011 |
|             | 409 |         |             | 100 |
|             | 517 |         |             | 101 |
|             | 616 |         |             | 110 |
|             | 713 |         |             | 111 |
|             | ... |         |             | ... |
o-------------o-----o         o-------------o-----o

Area codes (for the U.S. and Canada, in the early 1990's, before the
cell-phone population explosion had yet to reach its current force)
are constrained by an "intensional" rule to be elements of X that
may not have 0 or 1 as the first digit, that must have 0 or 1 as
the second digit, and that may have any numeral from 0 to 9 as
the third digit.  Thus, there are 8·2·10 = 160 possible codes
in the Area Code Table T and 1000 - 160 = 840 codes in the
complementary table X-T = X·(T).  As we let the universe
expand, like we have a choice, the size of T stays fixed
while (T) grows infinite in extension.  In other cases,
like T = People, both T and (T) remain somewhat vague
and indefinite in what they might be thought to hold.

Jon Awbrey
