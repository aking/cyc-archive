I feel that it is time for me to expand a little on my comments.

There are many issues about the engineering of a large system which are,
frankly, not ontology specific. Robert, above, is suggesting that we can somehow
use ontology structures themselves to do this engineering. This is very
dangerous territory:

a. In the software industry we already have a lot of experience in
componentizing and constructing large systems from small ones. This software
engineering `best practice' is ignored at our peril.

b. The focus of the SUO ontology should primarily be on the content of the
ontology itself; atempting to use ontology in this metalogical way is
distracting.

c. It is my belief that there a few core features that will enable proper
engineering of the SUO Ontology. However, these features will primarily be at
the representation language level. not within the Ontology itself. I.e., this
has more implications for KIF than for the SUO Ontology. For example, KIF has
inadequate mechanisms for combining sets of axioms. Personally, in this, I
prefer a semi-higher order approach (ala functional programming languages) over
a pure meta/text order apporach common in the logic programming community.
