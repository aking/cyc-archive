MW: I have developed (and given references to) similar documents for the
development of data models for similar purposes. When we undertook that work
in EPISTLE we concluded that it was better to produce those documents (the
map) before we undertook large scale modelling. It saved a great deal of
effort. Building models in KIF is not so different from doing so in an
Entity Relationship language. Most of the issues and principles stay the
same.

MW: However, you seem to have a very short memory, so I will repeat again a
list of document that make a better start point for discussion of what we
should do and how we should do it than just picking up some bits of ontology
and merging them together:

MW: ISO 18876-1 WD5 Integration of Industrial Data for Exchange Access and
Sharing: Overview and Architecture

MW: This document shows how a number of different models/ontologies can be
integrated. The methodology preserves the original model/ontology whilst
allowing the creation of an integration model/ontology. Different
integration models/ontologies can be created and linked by application
models/ontologies that are concepts that can be mapped into both
models/ontologies.

MW: ISO 18876-2 WD5 Integration of Industrial Data for Exchange Access and
Sharing: Integration and Mapping Methodology

MW: The methodology for how to operate the above architecture.

MW: "Some notes on the nature of things"

MW: An outline in plain language of a rationale for 4D a ontology/model.

MW: The above can be found at http://www.ISO18876.org if you dig around a
bit. All have been mentioned several times.

MW: "Developing High Quality Data Models V2"
http://www.stepcom.ncl.ac.uk/epistle/data/modelling/

MW: This document illustrates a number of traps that that those developing
data models frequently fall into when developing data models. It also
presents some principles for how to avoid these traps. In (at least) one
e-mail exchange here I have presented a version of these principles adapted
for KIF. They are not that complicated, but the current model does not
support them.

MW: EPISTLE Core Model V3.2 Revision 6 (aka ISO 15926-2 DIS draft)

MW: A data model which I posted in the last few days, part of which I
translated into KIF in November, and reposted again recently.

MW: So I really do not understand how you can accuse me of not presenting
the bits of the map I have. I only wish people would look at it, apparently
including you.

MW: I am not so arrogant as to suggest that the work mentioned above is
perfect in any sense. No map is. The point of a map is that it guides
travellers better than no map at all, and passes on the experience of other
travellers (or map makers).
